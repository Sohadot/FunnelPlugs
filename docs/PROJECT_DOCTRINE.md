# PROJECT_DOCTRINE.md

## 1) Status of the Asset

Funnelplugs.com is being built as a **sovereign-grade strategic digital asset**.

It is not being treated as a normal website, a lightweight affiliate property, or an SEO-first content machine.
It is being developed as a structured reference system with long-term acquisition value, recurring monetization potential, and institutional trust.

This project exists to transform a conceptually powerful domain into a durable digital asset that compounds in value.

---

## 2) Core Mission

The mission of Funnelplugs is to define, classify, and organize the logic of structural revenue leakage in digital customer journeys, then connect that logic to clear intervention layers capable of restoring integrity, continuity, and conversion flow.

The project is guided by the core narrative:

**The Logic of the Missing Plug**

Digital commerce does not fail only because it lacks traffic.
It fails because key structural plugs are missing inside the funnel.

Funnelplugs exists to make those absences visible and actionable.

---

## 3) Project Intent

The project is intended to achieve five things simultaneously:

1. Build a conceptually differentiated digital asset.
2. Establish a trusted reference position in a fragmented market.
3. Develop a governed system of vocabulary, standards, and classification.
4. Generate recurring income without degrading authority.
5. Increase long-term acquisition value through structure, trust, and utility.

---

## 4) What This Project Is Not

Funnelplugs must never drift into any of the following:

- a generic martech blog,
- an affiliate-first review site,
- a publishing volume machine,
- a trend-chasing SEO property,
- a visually premium but conceptually shallow brand shell,
- an ungoverned directory of software links.

Any move in these directions is strategic drift and must be corrected.

---

## 5) Project Mode of Operation

The project must be executed under a strict, sequential system.

### Required order of execution

1. Conceptual closure
2. Structural architecture
3. Owned vocabulary
4. Standards layer
5. Registry logic
6. Utility layer
7. Visual system
8. Automation
9. Controlled monetization
10. Scale and valuation maturation

No later layer may be built in a way that corrupts an earlier layer.

---

## 6) Publishing Doctrine

All project development occurs through the following sovereign chain:

**Local Python generation/update → GitHub Desktop commit/push → GitHub repository → GitHub Actions validation/build/deployment → live asset**

GitHub is the single source of truth.
Cloudflare exists only as the DNS, TLS, security, and edge-control layer after nameserver delegation.

No project publishing depends on Cloudflare API tokens.
No content or structural updates occur through Cloudflare.

---

## 7) Structural Goals

The project must gradually mature into the following layered system:

### 7.1 Conceptual Layer
The narrative and interpretive logic of the asset.

### 7.2 Protocol Layer
The methodological system that explains how the asset reads and classifies commercial leakage.

### 7.3 Standard Layer
The formal conditions that define structural integrity in digital customer journeys.

### 7.4 Registry Layer
The governed set of classified plugs, intervention categories, and evaluation structures.

### 7.5 Utility Layer
Tools, reports, or engines that translate the theory into practical decision support.

### 7.6 Trust Layer
Editorial discipline, consistency, evaluation clarity, and controlled monetization.

---

## 8) First Release Scope

The first release must remain controlled.
It must not launch bloated.
It must not imitate ordinary content sites.

The first release should revolve around five primary pages or equivalents:

- Manifesto
- Protocol
- Funnel Integrity Standard
- Plug Registry
- Revenue Leak Engine

These pages are the spine of the asset.
All later expansion must grow from them.

---

## 9) Project Discipline Rules

### 9.1 Every page must justify its existence
No page exists without structural purpose.

### 9.2 Every recommendation must be governed

### 9.3 Every monetization layer must protect trust

Scale without authority is strategic failure.
