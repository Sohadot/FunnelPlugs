System architecture

THE FUNNELPLUGS SYSTEM ARCHITECTURE

FunnelPlugs does not operate as a collection of disconnected pages. It operates as a governed diagnostic system for naming, classifying, and interpreting structural failure across commercial paths.

A governed system, not a generic CRO site

A generic conversion optimization site publishes articles, tips, and heuristics. Funnelplugs produces a different kind of output: governed failure language, structural classifications, diagnostic mappings, and intervention class references — grounded in a formal ontology, evaluated against an integrity standard, and processed through a defined protocol.

This distinction matters because diagnostic value depends on terminological stability. A practitioner who diagnoses "trust collapse" in a client's commercial path today must be working from the same structural definition as a practitioner who encountered the same condition six months ago. A system whose language drifts with each article it publishes cannot provide that stability. Funnelplugs is built to provide it.

How the layers connect

The Funnelplugs system is an ordered chain of interdependent layers. Each layer has a defined role and receives its inputs from the layer above it. Removing or destabilizing any layer degrades the diagnostic output of all layers below it.

The Manifesto establishes the founding position: revenue loss is often structural, not volumetric. This is not a content strategy claim. It is a categorical commitment that governs every downstream definition.

The Funnel Integrity Standard translates the founding position into formal integrity conditions: what a structurally complete commercial path requires across its trust, flow, and recovery dimensions. The standard is the evaluative benchmark. Without it, diagnostic output is impressionistic.

The Protocol defines how the standard is applied diagnostically. It specifies the detection, classification, and intervention assignment stages that convert an observed commercial path into a governed structural assessment.

The Registry organizes the intervention classes. Each registry category corresponds to a structural failure type and defines what must be present to restore the integrity condition the failure undermines.

The Engine operationalizes the full chain. It receives diagnostic signals from a practitioner evaluating a real commercial path and produces a governed output: classified failure types, integrity dimension assignments, and intervention priorities — grounded in the ontology and standard, not in heuristics.

The Reference layer makes every defined concept individually addressable. Each failure class and intervention class has a canonical URL, a stable definition, and internal links to related concepts. This layer serves human practitioners, search engines, and future systems that need to retrieve governed definitions without parsing unstructured prose.

The Ontology underlies all other layers. It is the controlled vocabulary — the formal names, identifiers, and definitions for every structural concept the system recognizes. When the ontology is stable, the system's diagnostic output is reliable. When the ontology drifts, every downstream layer becomes unreliable.

The Governance layer controls how all others change. It defines the change process, the versioning rules, the append-only record requirements, and the claims restraint policy. Governance is what makes the system durable — it is the mechanism that separates a category intelligence system from a content publication that happens to have good early structure.

How intelligence is produced from the system

Intelligence production in Funnelplugs follows a specific sequence. A practitioner evaluating a commercial path observes signals: friction patterns, trust failure indicators, flow interruptions, recovery absences. These signals are inputs.

The protocol maps signals to failure classes defined in the ontology. The standard determines which integrity dimensions are violated. The engine produces the structural assessment: which failure classes are present, in which dimensions, and which intervention classes are required to restore integrity.

The output is audit-ready. It is grounded in defined terms, evaluated against formal criteria, and structured for reproducibility. A second practitioner applying the same protocol to the same commercial path should reach the same structural classification.

This is different from the output of a generic optimization heuristic, which produces recommendations whose rationale is implicit and whose reproducibility across practitioners is low. Funnelplugs produces output whose rationale is explicit and whose reproducibility is structurally enforced by the protocol.

Why claims are restrained

Funnelplugs makes structural and diagnostic claims. It does not make revenue, ROI, or conversion-lift claims.

The engine classifies failure and assigns intervention classes. It does not predict what revenue change will result from applying an intervention. That prediction requires knowledge of execution quality, market conditions, competitive environment, and many other variables that are not observable from commercial path structure alone.

A system that makes financial predictions alongside structural classifications conflates two different knowledge domains. The structural classification can be grounded in observable path properties. The financial prediction cannot. Combining them in a single output degrades the credibility of both.

Funnelplugs therefore separates clearly what the system can determine — structural classification — from what it cannot — financial outcome. This restraint is not a limitation. It is a precision commitment. A precise claim about structure is more useful to a practitioner than an imprecise claim about revenue, because the practitioner can verify the structural claim and cannot verify the revenue claim without extensive additional work.

Why stable terminology matters

The value of a reference system is proportional to the stability of its terminology. A practitioner who spends time learning the system's classification framework is making an investment. That investment pays off if the terminology is stable — if the practitioner can apply what they learned six months later and still be working with accurate definitions.

Terminological instability has compounding costs. Every time a definition changes without documentation, a practitioner's prior knowledge becomes partially inaccurate. Every time a term is silently replaced with a synonym, internal consistency across the system's pages degrades. Every time a failure class is informally redefined, diagnostic outputs from different time periods become incomparable.

The ontology, the append-only policy, and the versioning policy exist to prevent this. They are not bureaucratic overhead. They are the mechanisms that make the system's terminology worth learning in the first place.

How the Engine applies the system without financial forecasts

The Revenue Leak Engine applies the Funnelplugs system through a structured diagnostic sequence. A practitioner answers a set of questions about the commercial path under evaluation. The engine maps responses to failure signals, classifies signals against the failure class ontology, evaluates the result against the integrity standard's dimensions, and produces a structured output.

The output states: which failure classes are present, which integrity dimensions they violate, which intervention classes are required, and what priority order addresses the most critical structural gaps first.

The output does not state: what revenue improvement will result, what conversion rate change should be expected, or what the return on investment of remediation will be. These are projections the system does not have the inputs to support. A practitioner who needs financial projections must construct them from their own data, informed by the structural diagnosis — not from claims made by the diagnostic system.

This architecture keeps the engine's output honest. Honest diagnostic output is more valuable over time than optimistic projection output, because practitioners learn to trust it.

How this architecture supports long-term category authority

Category authority is not built by volume alone. It is built by being the system that defines how a domain is understood. The system that establishes the terminology, the classification framework, the evaluation criteria, and the diagnostic method becomes the reference that other practitioners, systems, and publications work from or position against.

Funnelplugs is designed to be that system for structural funnel failure. The ontology names the failure types. The standard defines the integrity conditions. The protocol specifies the diagnostic method. The registry catalogs the intervention classes. The engine makes the system operational. The reference layer makes every concept discoverable.

As commercial systems become more automated and AI-assisted, the need for a stable, governed vocabulary for structural funnel failure increases. A system that can reliably classify failure types using defined language and formal criteria becomes increasingly valuable as the infrastructure that automated and human practitioners build on top of it grows.

The architecture is designed to remain legible — and authoritative — across that evolution.

Ontology class reference pages

The ten failure classes in the Plug Ontology are each documented as standalone reference pages. Each page covers the formal class definition, structural meaning, diagnostic logic, plug logic, standard dimension mapping, protocol stage mapping, engine integration, and registry integration for that class.

This reference layer materializes the ontology as individually addressable, internally linked, concept-depth pages. It converts the controlled vocabulary from a classification list into a navigable, practitioner-usable reference that supports diagnosis, learning, and systematic path evaluation. The ten pages are accessible from the Ontology page and the Reference Index.
