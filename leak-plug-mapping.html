Leak-to-plug mapping

Diagnosis produces a failure class. That class maps to a specific intervention class. The relationship is governed, not guessed.

Scope

This concept defines the formal correspondence system between structural failure classes (leaks) and structural intervention classes (plugs). Leak-plug mapping bridges diagnostic language and intervention language without collapsing them — preserving the distinction between identifying a failure and specifying its structural remedy. It governs application of the Plug Ontology in intervention decisions.

What leak-plug mapping is

Leak-plug mapping is the governed correspondence system that defines which structural intervention class (plug) addresses each structural failure class (leak). It is not a recommendation engine. It is not a best-practices list or an optimization checklist. It is the formal specification of cause-and-response relationships within the Plug Ontology — the authoritative source for which structural intervention is appropriate for each identified failure class.

The distinction between diagnostic language and intervention language is maintained deliberately. A leak class describes what is structurally absent or broken — the nature of the failure, its position in the path, and its behavioral consequences. A plug class describes the structural intervention required to address that failure — the mechanism, its installation position, and its relationship to adjacent structural elements. These are distinct vocabularies that must not collapse into each other. Recommending a plug without a corresponding leak class diagnosis is structural guessing. Diagnosing a leak class without a governing plug class mapping leaves the diagnosis without an actionable intervention specification.

Leak-plug mapping preserves the asymmetries of real structural failure. The relationship between leak classes and plug classes is not uniformly one-to-one. Some leak classes require multiple plug interventions because a single structural failure can manifest across multiple path positions and require layered responses. Some plug classes address multiple leak class conditions because certain structural elements resolve failures that share a common upstream cause. The map governs these relationships formally — capturing the complexity of real commercial path architecture rather than simplifying it for presentation convenience.

The map is the authoritative reference when diagnostic output produces a failure class identification and intervention specification is required. It is also the governing constraint on what interventions are valid for a given diagnosis — preventing the installation of plugs that have no correspondence to the identified failure class.

The mapping logic

Each plug class exists because a corresponding leak class can occur. This is the foundational logic of the Plug Ontology: the taxonomy of interventions is derived from the taxonomy of failures, not from a separate classification of marketing techniques or optimization methods. If a leak class exists in the ontology, its corresponding plug class exists. If a plug class is identified as relevant, it is because a diagnosed leak class maps to it.

The mapping logic operates in three relationship types. The first is direct correspondence: one leak class maps to one primary plug class, and the intervention fully addresses the failure when correctly installed. Trust Collapse maps directly to Trust Infrastructure — the plug addresses the exact structural absence that the leak class identifies. The second relationship type is compound response: one leak class maps to multiple plug classes because the failure has multiple structural manifestations. Friction Overload, for example, may require both a Friction Reduction Layer at the specific dropout point and a Flow Continuity Layer that governs the full step sequence — because localized friction is often symptomatic of a broader flow architecture absence.

The third relationship type is shared response: multiple leak classes map to the same plug class because the structural absence is common to both failures. Certain alignment failures — Message-Path Misalignment and Audience Qualification Failure — both indicate deficits in the Alignment Reinforcement Layer, even though they differ in their diagnostic origin. The map governs all three relationship types, ensuring that intervention specification is accurate regardless of relationship complexity. The Reference Index provides the full classification vocabulary used in mapping decisions.

Core leak-plug pairs

Trust Collapse → Trust Infrastructure. Trust Collapse is identified by active engagement at the offer stage with suppressed conversion — the buyer arrives but declines because trust signals are absent or insufficient. The Trust Infrastructure plug class addresses this by installing the structural elements that make offer acceptance credible: social proof architecture, authority signal placement, risk reversal mechanisms, and commitment escalation pathways. The relationship logic is direct: the leak is a trust deficit; the plug is trust construction.

Friction Overload → Friction Reduction Layer. Friction Overload is identified by concentrated dropout at specific path positions — form stages, checkout steps, or decision-point pages where buyer effort exceeds buyer motivation. The Friction Reduction Layer addresses this by restructuring the friction-producing elements: simplifying form architecture, reducing commitment thresholds, removing unnecessary steps, and improving cognitive load management at high-friction positions. The relationship logic is structural: the leak is excess resistance; the plug is resistance removal.

Flow Break → Flow Continuity Layer. Flow Break is identified by interruption of forward path progression that is not attributable to friction or trust failure — technical breaks, navigation discontinuities, or sequence gaps that prevent buyers from completing the intended journey. The Flow Continuity Layer addresses this by installing the structural elements that maintain path integrity: clear progression architecture, step continuity signaling, and fallback path routing for disrupted sequences.

Recovery Absence → Recovery Systems Layer. Recovery Absence is identified by post-abandonment exit with no re-entry signal — permanent loss of buyers who exited the path before completing the intended action, with no recovery mechanism to re-engage them. The Recovery Systems Layer addresses this with post-exit re-engagement sequences, abandoned session recovery architecture, segmented re-entry pathways, and timing-governed re-contact protocols. The relationship logic is temporal: the leak is unrecovered exits; the plug is recovery infrastructure that converts exits into delayed completions.

Decision Ambiguity → Decision Stage Integrity. Decision Ambiguity is identified by extended dwell time with no forward progression at decision points — buyers who are present but structurally unable to commit because the decision architecture is absent or unclear. The Decision Stage Integrity plug addresses this by installing clarity mechanisms: comparison architectures, consequence articulation, friction-free commitment pathways, and decision scaffolding that reduces cognitive resistance to the act of choosing.

Message-Path Misalignment → Alignment Reinforcement Layer. Message-Path Misalignment is identified by high entry rates with rapid early dropout — traffic that arrives under one expectation and encounters a path designed for a different expectation. The Alignment Reinforcement Layer addresses this by restoring narrative consistency between the entry-stage message and the path's offer architecture: entry signal continuity, expectation-setting elements at path entry, and offer framing that validates the arriving buyer's pre-existing intent.

Using the map for diagnostic priority

When multiple leak classes are present simultaneously — which is the common condition in structurally incomplete commercial paths — leak-plug mapping governs intervention sequencing. The map is the reference for determining which plug must be installed first, which has upstream dependencies that must be resolved before installation is meaningful, and which can be installed in parallel without creating structural conflicts.

The governing principle is structural dependency: plugs whose absence prevents other plugs from operating must be installed first. A Recovery Systems Layer that is installed before Trust Infrastructure is in place will generate re-engagement sequences that reach buyers who were never given adequate trust foundation — the recovery effort engages buyers who will decline again for the same reason they declined initially. The map identifies this dependency relationship explicitly, preventing the resource waste of out-of-sequence installation.

Entry-level plugs (Alignment Reinforcement, Audience Qualification) govern before conversion-level plugs because they determine whether qualified buyers reach the conversion stage. Trust Infrastructure governs before conversion optimization because conversion mechanisms depend on trust infrastructure being present. Recovery layer installation follows initial path stabilization because recovery requires a functional primary path to return buyers to. The map makes these dependencies traceable and the sequencing logic auditable.

The registry and the map

The Plug Registry operationalizes the leak-plug map by cataloging all governed plug classes with their full specification: structural definition, installation position, dependency relationships, and behavioral outcome criteria. The Registry is the reference layer that translates the map's class-level correspondences into actionable plug specifications for a given path. The Plug Ontology is the classification authority — the source of the leak and plug class vocabularies that the map uses. The Reference Index provides cross-access to all classification, mapping, and specification content in the Funnelplugs system. The Revenue Leak Diagnostic Engine applies the map's correspondence logic as a structured workflow, accepting diagnostic inputs and producing plug class recommendations governed by the mapping relationships defined here.
