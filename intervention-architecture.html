Intervention architecture

A governed system design in which every intervention layer is selected, positioned, and sequenced by the failure class it addresses — not accumulated without structural logic.

Scope

Intervention Architecture defines the complete structural design of intervention layers within a commercial path as governed by the Plug Ontology. It distinguishes deliberate, classification-driven architecture from the ad hoc accumulation of tools and tactics that characterizes most funnel build approaches.

What Intervention Architecture Is

Intervention Architecture is the complete, deliberate structural design of intervention layers within a commercial path. The defining characteristic is deliberateness: each intervention layer within the architecture is selected because a specific classified failure class requires it, positioned where that failure class operates, and sequenced relative to other layers based on structural dependencies between the failure classes they address.

Intervention Architecture is not a checklist of best practices. Best practices are decontextualized recommendations that do not account for the specific failure class profile of a given commercial path. Intervention Architecture is context-specific: it is derived from a diagnosis of the path's actual failure class distribution and governed by the structural logic that maps each failure class to its correct intervention response.

This distinction has direct practical consequences. A path built from best practices accumulates interventions without governing logic — each addition is locally justified but the system as a whole lacks structural coherence. Interventions may address failure classes that are not present in the path while leaving classified failures that are present unaddressed. The intervention collection grows but the structural completeness of the path does not improve proportionally, because no governing logic ensures that the additions are targeted at the actual failure conditions.

Intervention Architecture solves this by subordinating every architectural decision to a prior classification: before any intervention is selected, the failure class it addresses must be identified, confirmed as present in the path, and assigned a priority within the overall failure class profile. Only then is the intervention layer designed and positioned. The result is a path in which every structural element exists for a specific, governed reason.

Architecture vs. Collection

The distinction between intervention architecture and intervention collection is among the most consequential in commercial path design, and it is among the least recognized. Most funnel builds produce intervention collections. Intervention architecture is rare because it requires a prior classification system — the Plug Ontology — to govern it.

An intervention collection is the product of additive decision-making without governing logic. A tool is added because it is available. A tactic is implemented because it performed for a different path or industry. A layer is included because it is considered best practice in the current period. Each addition is independently justified; none is governed by a prior diagnosis of the specific failure the path presents. Over time, the collection grows. The path acquires more elements but not more structural coherence.

The failure mode of intervention collections is predictable: they address symptoms rather than classified structural causes, they may address failure classes not present in the path while leaving present failure classes unaddressed, and they accumulate friction and complexity that can suppress conversion at stages where the structural failure has already been resolved. A funnel with 20 tools and no governing architecture is still structurally incomplete if the classified failures present in the path are not specifically addressed by those tools.

Intervention Architecture inverts this logic. A funnel with six precisely positioned interventions governed by the plug classification system can achieve structural completeness if those six interventions address the six failure classes diagnosed as present in the path. Structural completeness is not a function of intervention count — it is a function of the match between the failure class profile of the path and the intervention layers selected to address it. Architecture thinking produces this match; collection thinking cannot.

The Four Architecture Layers

A structurally complete commercial path requires four intervention architecture layers. The layers are not sequential stages — they operate in parallel across the path — but they address distinct failure class categories and require distinct design logic.

Entry architecture encompasses the intervention layers that capture and direct buyer intent at the point of path entry. Entry architecture addresses acquisition-stage failure classes: the misalignment between acquisition message and landing experience, the failure to segment incoming intent by source and readiness level, and the loss of intent at the first path transition. Entry architecture interventions include message-aligned landing experiences, source-specific entry paths, and intent-capture mechanisms positioned at path entry. A path without deliberate entry architecture loses intent before downstream intervention layers have an opportunity to operate.

Conversion architecture encompasses the intervention layers at commitment-required stages: trust infrastructure, offer clarity mechanisms, commitment term disclosure, friction-removal systems, and single-action commitment design. Conversion architecture addresses decision-stage failure classes — Decision Ambiguity, Trust Deficit, Offer-Action Mismatch — and must be calibrated to the specific commitment level of each decision stage in the path. This is the layer most commonly addressed by ad hoc optimization work and most commonly left structurally incomplete because optimization work is applied before the structural requirements are met.

Recovery architecture encompasses the intervention layers that re-engage buyers who have exited the path without converting. Recovery architecture addresses exit and abandonment failure classes: buyers who exit at specific stages, buyers who initiate commitment and abandon before completing, and buyers who were interested but not yet ready at the point of first contact. Recovery architecture includes abandonment sequences, retargeting frameworks, and re-entry path design. A path without recovery architecture forfeits all buyers who exit before conversion; recovery architecture converts a portion of this exit population.

Intelligence architecture encompasses the signal collection, measurement frameworks, and diagnostic feedback loops that enable ongoing identification and correction of structural failures. Intelligence architecture is the layer that makes the other three layers improvable over time: without systematic signal collection and diagnostic feedback, structural failures that emerge or evolve cannot be identified and corrected. Intelligence architecture includes event tracking at each decision-critical position, segment-specific conversion measurement, and structured diagnostic review cycles that apply the plug classification framework to emerging behavioral data.

Architecture Governance

The Plug Ontology governs intervention selection within the architecture. This governance function is what distinguishes Intervention Architecture from intervention collection and what prevents the accumulation of structurally incoherent additions over time.

Governance operates through classification: before any intervention layer is added to or modified within the architecture, the failure class it addresses must be classified using the Plug Ontology's 10-class framework. The classified failure class determines which architecture layer the intervention belongs to, where within that layer it must be positioned, and what structural design requirements it must meet. An intervention that does not map to a classified failure class present in the path is not included in the architecture — regardless of its performance in other contexts or its status as an industry best practice.

This classification requirement prevents two failure modes of unmanaged architectures: over-addition, in which interventions accumulate without a removal mechanism for elements that do not address present failure classes; and mis-sequencing, in which interventions are positioned at stages where their target failure class does not operate, while the stages where that failure class is active remain unaddressed. The Funnelplugs Protocol specifies the governance process for architectural decisions, and the Funnelplugs Standard defines the structural completeness thresholds that a governed architecture must meet across each layer.

The Engine Operationalizes This

The Revenue Leak Diagnostic Engine applies intervention architecture logic interactively to a specific commercial path. Rather than presenting intervention architecture as an abstract framework, the Engine walks the diagnostic process — classifying failure conditions present in the path, mapping each classification to the appropriate architecture layer, and producing a governed intervention specification that addresses the actual failure class profile of the path being analyzed.

The Engine's output is a path-specific intervention architecture governed by the Plug Ontology — not a generic recommendation set, but a classified, positioned, and sequenced intervention design derived from the specific structural failures diagnosed. The Plug Registry provides the complete reference catalog of classified interventions from which the architecture is assembled.
